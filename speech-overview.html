01 Evolution

AI 编程的三段演化

今天的智能体能力，建立在前两个阶段长期积累起来的上下文理解、交互方式和工程视野之上。

01

代码补全时代

以 Tabnine、GitHub Copilot 为代表。核心能力是局部补全，对项目整体几乎没有理解。

关键词：局部预测、机械输入、单文件上下文。
02

对话副驾驶时代

Cursor、Trae 这类工具开始理解项目上下文，但结果质量仍然高度依赖人的判断和审查。

关键词：项目级视野、对话协作、人工 review。
03

智能体工作流时代

Claude Code、Codex、OpenCode 把“计划、实施、验证”串成闭环，AI 开始真正参与执行。

关键词：任务闭环、主动调用工具、结果验证。
02 Stage 3 Focus

阶段 3 的意义，不在概念本身，而在它已经进入实际工作流。

和前两个阶段相比，这里的变化是：AI 不再只生成内容，而是开始围绕目标完成任务。

更长上下文 更强专业能力 更低成本 多模态 + 工具链

浏览器里复现并修 BUG

给出复现路径后，智能体可以自己操作页面、定位问题、修改代码，再回头验证。

把 Markdown 直接做成 PPT

给它一篇文档或演讲稿，它可以直接整理成演示稿，并把版式统一起来。

在真实渲染结果上做 UI

它可以基于真实页面的渲染结果，而不是只看静态代码，持续调整界面细节。

沿着工程体系做国际化

它可以沿着现有的 i18n 和工程结构，把文案翻译后准确落回对应位置。

03 Infrastructure

MCP 决定它能做什么，Skills 决定它该怎么做。

比模型接入更重要的，是一套能复用、能约束、能验证的工作流。

MCP

决定 AI 能读什么、操作什么、验证什么，是能力边界。

Skills

决定 AI 该按什么原则做事，是行为约束。

MCP Examples

常用 MCP

chrome-devtools

直接操作浏览器，查看真实渲染结果，并读取 Console、Network 等调试信息。

Take snapshot 把复杂 DOM 压缩成更适合 AI 决策的语义结构。
Take screenshot 直接基于真实页面截图继续判断和迭代。
Emulate 切换设备、主题和网络条件，检查多场景表现。
Evaluate Script 执行脚本、拿运行态数据、进入浏览器调试现场。
fetch

拉取网页和文档内容，适合查询官方资料、API 说明和错误信息。

context7

补充最新文档和框架知识，适合处理依赖官方文档上下文的开发任务。

数据库操作

直接查询或修改数据库，适合排查数据问题、验证状态和处理后台联动场景。

Skills Examples

常用 Skills

frontend-design

补充界面设计方向和视觉约束，适合处理布局、风格和组件表现相关的任务。

find-skills

帮助 AI 快速找到合适的技能集合，减少在能力选择上的试错成本。

upload-images

把本地图片转成可访问的 URL，方便把截图、设计稿或素材交给其他模型继续处理。

iconify

统一图标选型、接入方式和渲染细节，减少实现过程中的碎判断。

create-skills

辅助创建自定义技能，定义智能体的行为约束和工作流规范。

karpathy-guidelines

将 Karpathy 四原则嵌入协作流程，确保智能体按工程规则推进任务。

04 Tooling

CC Switch

配置智能体的超实用必备应用。

集中管理供应商、MCP、Skills、提示词

把能力配置、行为约束和调用入口收拢到同一个工作台里统一维护。

支持 API 代理格式转换与故障转移

在不同模型供应商之间做协议适配，并在服务异常时平滑切换链路。

支持配置导入、导出、备份、同步

本地迁移和团队共享都更直接，不会把配置资产困在单台机器里。

05 Method

Karpathy 编程四原则

它们不是提示词技巧，而是适合智能体协作的工程规则。

编码前思考

把假设和疑问先说清楚，避免一开始就站在错误前提上。

简洁优先

能用更直接的方案解决，就不要引入多余抽象。

精准修改

只改和目标直接相关的部分，减少噪音和连带风险。

目标驱动执行

先定义什么算成功，再围绕验证结果推进实现。

前端工程师的核心价值，正在从“写得快”转向“判断得准、约束得住、验收得清”。 代码会越来越便宜，但目标定义、流程设计、验证标准和结果判断，会越来越重要。
