# AI / LLM / 智能体

## 面向前端团队的一次内部分享

大家好，今天这场分享我想聊的主题，是 AI、LLM 还有最近大家已经高强度使用的智能体。

这不是一场“宏大叙事”式的趋势报告，也不是一场工具测评合集。我更想从一个一线前端工程师的视角，聊聊这两年我真实感受到的变化：AI 编程到底经历了哪些阶段；为什么很多人会突然觉得“今年的 AI 跟去年的 AI 不是一个东西”；以及对于我们前端工程师来说，应该怎样理解、使用和驾驭这些能力。

如果把今天的内容压缩成三个问题，大概是：

1. AI 编程是怎样一步步演化到今天这个阶段的？
2. 智能体到底“强”在什么地方？它和普通对话式 AI 的区别是什么？
3. 在实际工程里，我们应该怎样跟 AI 协作，才能既提效，又不失控？

接下来我会按照这个顺序展开。

---

## 一、AI 编程的进化路线图

我自己理解，AI 编程的演化，至少可以分成三个阶段。

### 1. 代码补全时代

第一个阶段，大家都非常熟悉，就是代码补全时代。

这个阶段的代表工具，主要是以 VS Code 插件形态出现的，比如 Tabnine、Github Copilot。它们最典型的使用方式就是：你写到一半，AI 给你补几行代码，你按一下 Tab 接受建议。

这个阶段的核心特征，是“局部预测”。它更像一个基于概率的字符串拼接机器。它能根据你当前文件、当前函数、当前光标附近的上下文，去推测你下一步大概率会写什么。

举个最简单的例子：你在代码里输入 `function debounce(`，AI 可能马上就帮你把整个 debounce 函数补完。这种体验在当时其实已经很震撼了，因为我们第一次非常直观地感受到，大模型不是只能聊天，它是真的能参与编程。

但这个阶段的边界也很明显。它本质上还是“看局部、补局部”。它对项目全局几乎没有理解，不知道你的业务背景，不理解工程架构，也不知道你这次修改最终要落到什么目标上。它更像一个非常聪明的输入法，而不是一个真正意义上的协作者。

所以在这个阶段里，人类的角色仍然是开发工程师本身。我们需要对整个仓库有完整的理解，知道模块之间怎么关联，知道为什么要这么写，也知道哪些代码不能碰。AI 更多只是帮我们省掉一部分机械性的键盘输入。

但是必须承认，正是从这个阶段开始，我们第一次从“原始编程时代”进入“AI 编程时代”。那种“AI 好像真的懂我想写什么”的感觉，是很多人第一次被大模型打到的时刻。

### 2. 对话副驾驶时代

第二个阶段，我把它称为“对话副驾驶时代”。

这个阶段的代表工具，通常是 VS Code 插件，或者独立 IDE，比如 Cursor、Trae 这一类。和上一阶段相比，最大的变化不是“补得更快”，而是 AI 开始摆脱单文件限制，拥有项目级的工程视野。

你不再只是等它补全一行代码，而是可以把报错信息贴给它，让它分析问题；你可以把需求文档、关键文件、代码片段，甚至一张设计图、一张报错截图扔给它，然后让它帮你实现功能、定位问题、补测试、写重构方案。

从这个阶段开始，提示词开始变得重要，MCP 和 Skills 这样的概念也逐渐进入视野。因为大家慢慢发现，AI 的输出效果不只是由模型本身决定，也高度取决于你怎样给上下文、怎样描述目标、怎样约束它的行为。

但这个阶段依然远远没有到“放心交给它”的程度。我们会明显感觉到，AI 虽然比以前懂得更多了，但它依然很容易在细节上出错，也经常在上下文不足时一本正经地写错代码。于是人在这个阶段里的角色，就从“开发工程师”慢慢变成了“开发工程师 + 代码审查员”。

这也是我自己在使用国内版 Trae 时非常直观的感受。那时候用的是字节自家的免费模型，确实能帮我产出不少代码，但几乎每次生成后我都要认真 review，而且 review 完往往还要自己再手动调整。所以那个阶段我对 AI 编程的信心其实并不强，我会把它当成副驾驶，但不会把方向盘交给它。

### 3. 智能体工作流时代

第三个阶段，就是现在大家真正开始兴奋起来的阶段，也就是智能体工作流时代。

这里说的 AI Coding Agent，本质上就是 AI 编码智能体。它和前两个阶段最大的差别，不是“回答更聪明一点”，而是从“被动响应”变成了“自主行动”。

为什么这个阶段突然爆发？我觉得背后有几个很现实的原因：

1. 大模型支持的上下文越来越大，不再只能盯着几百行代码做判断。
2. 模型的专业能力明显增强，对代码、架构、文档和调试的理解都上了一个台阶。
3. 模型价格下降得很快，调用成本越来越可接受。
4. 主流模型基本都支持多模态，文本、图片、网页、终端、浏览器这些输入输出可以被放到同一个工作流里。

所以这一阶段的代表工具，也开始从“对话型 IDE”升级为真正的 Agent 工具，比如 Claude Code、Codex、OpenCode。

它们的核心特征是：为了完成任务，它不只是回答你一句话，而是会主动制定计划、调用工具、翻阅文档、操作浏览器、修改文件、运行验证，然后把整个“计划 - 实施 - 验证”的闭环走完。

这件事一旦发生，工作方式就变了。

以前我们会说：“帮我生成一个函数。”

现在我们更可能说：“这个 BUG 的复现路径是这样的，你去浏览器里复现，找到根因，改掉，跑验证，再告诉我结果。”

以前我们会说：“帮我写几页 PPT 大纲。”

现在我们可以直接说：“这里有一篇 Markdown，帮我做一份完整 PPT，统一样式，检查排版，再导出成演示稿。”

对于前端团队来说，这个变化尤其明显。因为前端天然就和浏览器、多端适配、视觉效果、交互行为紧密相关。智能体如果能直接打开页面、截图、切换亮暗色、模拟移动端、查看 Console 和 Network，再结合设计规则去调整 UI，它对前端开发流程的渗透会非常深。

比如这些场景，在今天都已经非常真实：

- 描述 BUG 复现路径，智能体自动在浏览器中操作页面、定位问题、修改关键代码并验证结果。
- 提供一篇 Markdown，让智能体自动生成 PPT 或文档型输出。
- 让智能体在亮色、暗色、PC、移动端之间切换，基于真实渲染结果去设计和微调 UI。
- 基于项目架构和已有国际化体系，把一篇中文文档翻译成其他语言版本。

这些都说明，Agent 已经不是“写代码的聊天机器人”，而是在朝“可执行的软件协作者”演化。

## 二、从 OpenCode 到多智能体协作

我接触的第一个编程智能体，其实是 OpenCode。倒不是因为它本身有多么决定性，而是因为我当时在 X 上看到很多人在分享一个插件，叫 `oh-my-opencode`。

### 1. oh-my-opencode 为什么让人印象深刻

这套东西很有意思。它本质上是一款搭配 OpenCode 使用的多模型智能体编排框架。

它不是把所有问题都丢给一个大模型，而是内置了多个“专职智能体”，类似 Claude Code 里的 subagents，每个角色只负责自己擅长的事情：

- **Sisyphus（西西弗斯）**：主架构师，负责任务拆解和全局把控。
- **Librarian（图书管理员）**：负责看文档、找上下文、补充信息。
- **Oracle（神谕）**：负责 Code Review，只挑问题，不写代码。
- **Frontend Engineer**：专门负责切图、视觉还原和 UI 落地。

这套设计最打动我的地方，不是“角色扮演”这件事本身，而是它背后的思路非常工程化：不同的任务，不一定要交给同一个模型去做。

比如最强最新的 GPT 或 Claude 旗舰模型，可以用来做架构判断和关键决策；一些更便宜但检索和阅读能力不错的模型，可以用来做文档理解；而在视觉和前端还原方面表现更好的模型，又可以用来负责 UI 任务。

也就是说，它不是简单追求“单模型最强”，而是在追求“整体工作流更合理、成本更可控、结果更稳定”。

它后来会爆火，我觉得原因也很典型：

1. 它被 Anthropic 点名封杀，这本身就很有传播性。
2. 它把“不同模型做不同任务”这件事讲清楚了，也让大家第一次对成本优化有了直观感受。
3. 它开箱即用，内置了很多好用的 MCP、Skills、提示词，以及配套工具链，让使用门槛大幅下降。

### 2. CC Switch 这类工具为什么重要

除了 Agent 本体之外，我还想提一个很实用的配套工具，就是 CC Switch。

这类工具的重要性在于，它不是直接替你干活，而是在帮你管理“智能体运行的环境”。

比如它可以集中管理模型供应商、MCP、Skills、提示词；可以做 API 代理格式转换和故障转移；也支持配置导入、导出、备份和同步。

这类能力乍一看不像“核心功能”，但当你真的开始长期使用多个模型、多套配置、多种工作流时，你会发现它非常关键。因为智能体真正进入日常工作之后，工程师面对的不再是“一个聊天窗口”，而是一整套需要维护的 AI 工具链。

### 3. 一个现实问题：token 消耗会越来越大

这里也顺带说一个很现实的问题。像 OpenClaw 这类使用体验跑下来之后，我有一个非常强烈的感受：token 消耗量会持续飙升。

原因很简单。因为 Agent 不是只回你一句话，它会看更多文件、读更多文档、跑更多轮验证，还会不断把中间结果带回上下文继续推理。上下文一长、步骤一多、工具一开，token 的消耗就不是聊天助手时代那个量级了。

所以如果未来公司内部真的开始大规模使用 Agent，模型效果、工具能力、成本控制，这三件事一定是绑在一起讨论的。

## 三、MCP 与 Skills：Agent 的两套基础设施

如果只让我用一句话概括 MCP 和 Skills 的关系，我会这么说：

MCP 是 AI 的手和脚，决定它能接触到什么、能操作什么；
Skills 是 AI 的脑回路和专业知识，决定它应该怎样思考，以及以什么风格做事。

这两者缺一不可。

### 1. 什么是 MCP

MCP 本质上是一个标准通信协议。它允许智能体向外部环境请求数据，或者执行动作。

它是双向的、可执行的、动态的。只要你把某种能力通过 MCP 暴露出来，智能体就不再只是“知道”，而是“真的能做”。

比如它可以去读网页、操作浏览器、访问数据库、读取本地文件、执行终端命令，或者跟其他外部系统交互。

### 2. 为什么我会重点讲 chrome-devtools-mcp

在各种 MCP 里，我觉得对前端团队最值得重点关注的一个，就是 `chrome-devtools-mcp`。

它做的事情很直接：让大模型通过 MCP 与 Chrome DevTools Protocol，也就是 CDP 通信。

一旦打通这层能力，智能体就获得了非常像“真人操作浏览器”的能力。它可以观察网页、理解页面结构、点击元素、执行脚本、查看渲染状态，还能直接拿到 Console 和 Network 面板的数据。

这意味着它不再只是“根据代码猜页面长什么样”，而是可以真的打开页面看。

对前端来说，这个差异非常大。因为很多问题本来就不是单纯读代码能判断出来的，必须结合真实渲染、真实交互、真实响应去看。

#### Take snapshot

`Take snapshot` 对我来说是一个非常有代表性的能力。

它背后调用的是 CDP 提供的 `DOMSnapshot.captureSnapshot`。这个能力会把 HTML 里冗长、复杂的 DOM 树，压缩成一棵对 AI 更友好的语义树。返回的信息不再是原始 HTML 的噪音，而是高度提炼过、适合决策的页面结构信息。

比如 AI 判断要点击一个按钮，它不一定需要理解整份 HTML，只需要根据快照里元素的语义和 uid，给出像下面这样的操作指令：

```jsx
{
  "server": "chrome-devtools",
  "tool": "click",
  "arguments": {
    "uid": "10_34",
    "includeSnapshot": true
  }
}
```

这个过程很像人类先“看一眼页面结构”，再决定下一步点哪里。

#### Take screenshot

`Take screenshot` 则更直观。它可以获取当前视窗截图、某个 DOM 元素截图，或者整张网页截图。

这对于前端特别重要，因为很多问题压根不是逻辑错，而是视觉错、间距错、响应式错、主题切换错。能截图，就意味着 AI 可以基于结果继续迭代，而不是停留在猜测层面。

#### Emulate

`Emulate` 能模拟移动端和 PC 端，模拟亮色和暗色模式，甚至模拟网络限制。

这意味着我们可以要求智能体不是只看“我本机当前这个页面”，而是系统性地检查多个场景下的表现是否一致。这对设计语言统一、多端兼容、主题适配，都很有价值。

#### Evaluate Script

`Evaluate Script` 则提供了一个更强的入口：可以直接执行任意 JavaScript。

你可以让它拿任意数据、获取组件实例、拿到 DOM 或 Window 对象引用、拦截网络请求、批量操作节点。某种意义上，这相当于让 Agent 直接进入了浏览器调试现场。

### 3. 什么是 Skills

如果说 MCP 解决的是“能不能做”，那 Skills 解决的就是“应该怎么做”。

Skills 本质上是高级的、结构化的上下文注入。你在和智能体对话时唤醒一个 Skill，本质上就是把一段预设好的专家级提示词、行为准则、规则说明塞进模型的系统提示词中。

它不是动态调用工具，而是静态约束模型的思考方式和执行风格。

比如你让智能体设计一个精美的登录页，如果只有 `frontend-design` 这样的 Skill，没有浏览器 MCP，它可能能做出一个“局部看着还行”的页面，但整体比例、层级、留白和真实渲染效果可能很糟。

反过来，如果只有 `chrome-devtools` 这样的 MCP，没有设计类 Skill，它虽然能打开页面、截图、验证，却不知道什么叫“好看”，也不知道该遵循什么设计语言。

所以对 Agent 来说，真正有价值的不是 MCP 和 Skills 二选一，而是两者配合：

- Skill 负责定义设计原则、工程规范、行为边界。
- MCP 负责提供观察、操作、验证的能力。
- 两者结合之后，智能体才有机会既“知道要做什么”，又“真的把事情做出来，并验证结果”。

这也是为什么我越来越觉得，未来团队里真正的 AI 基础设施建设，重点不是“买一个最强模型”，而是把 MCP、Skills、工作流、验证机制整合好。

## 四、AI 编程时的基本准则

讲完工具和工作流，我想回到一个更关键的问题：我们应该怎样使用这些能力？

因为工具再强，如果使用方式不对，最终得到的也只是一个高速制造错误的系统。

结合这段时间的使用经验，我自己有几个非常朴素但很有效的原则。

### 1. 说出来，让 AI 真的懂你

很多时候 AI 效果差，不是因为模型不行，而是因为我们把关键信息省略掉了。

如果你脑子里已经有判断标准、有业务边界、有不希望它碰的文件、有必须经过的验证步骤，那这些都不要藏在心里，要明确说出来。

越是复杂任务，越不能靠 AI 猜。因为人类能靠经验补全上下文，模型不行。

### 2. 拆解自己的工作流程，交代给 AI

很多工程师会给 AI 一个抽象目标，比如“把这个问题修掉”。这通常不够。

更有效的方式是，把你平时自己干活时的思考过程拆开来交给它。比如：

1. 先理解问题和影响范围。
2. 再定位相关代码和上下文。
3. 再提出修改方案。
4. 修改完成后跑验证。
5. 最后再回头检查是否引入副作用。

当你把工作流显式描述出来，Agent 的表现通常会稳定很多。因为它不再只是生成答案，而是在执行一个你定义好的过程。

### 3. 智能体一定要支持在线搜索

如果 Agent 不能在线搜索，它的上限会低很多。

原因很简单：很多问题本身就依赖最新信息，比如库的版本变化、官方文档更新、兼容性说明、社区 issue、浏览器行为差异。这些内容一旦过期，AI 就会非常自信地给你一个旧答案。

尤其是前端生态变化快，框架、构建工具、浏览器标准都在持续变。没有在线搜索能力的智能体，很多时候只是“记忆里的专家”，而不是“当前环境里的专家”。

### 4. 遇到 UI、交互、报错类问题，尽量给图片

这一点我很想强调。

前端里有很多问题，本来就是视觉和上下文敏感的。你如果只发一句“这个页面不好看”“这个弹窗位置不对”“这个报错很奇怪”，AI 能理解到的其实非常有限。

但如果你把截图、设计稿、浏览器报错截图、页面现状一起给它，它的判断质量会明显上升。尤其是在多模态能力已经普及的今天，图片本身就是非常高价值的上下文。

## 五、karparthy 编程四原则，以及它为什么适合 Agent

我很喜欢 Andrej Karpathy 提出的那套编程原则，因为它本质上不是在教人“怎么写提示词”，而是在教你如何把工程上的好习惯，翻译成 Agent 能执行的规则。

我把它理解成四句话。

### 1. 编码前先思考

不要假设，不要妄下断言，不要隐藏困惑，要把自己的权衡讲清楚。

这件事对 Agent 尤其重要。因为模型一旦基于错误假设启动，它会非常努力地在错误方向上一路做下去，而且做得看起来还挺像那么回事。

所以更好的方式是：

- 明确陈述假设。
- 如果存在多种解释，就先提出，不要偷偷选一种。
- 如果有更简单的方法，要主动说明。
- 如果关键信息缺失，要先问清楚或者先验证。

这会显著减少“看似勤奋，实则跑偏”的情况。

### 2. 简洁优先

用最少的代码解决问题，不做没有被要求的抽象，不提供没有被要求的灵活性，也不要为几乎不可能发生的情况设计一堆复杂逻辑。

这条原则非常适合拿来约束 AI。因为 AI 很容易犯的一个典型错误，就是“过度用力”。明明几十行代码能解决的问题，它可能给你写成两百行，再顺手加上配置项、扩展点、工厂模式和一套你根本用不上的抽象。

所以判断标准可以非常简单：如果一个资深工程师看了会觉得“这也太复杂了”，那就大概率真的复杂了。

### 3. 精准修改

只碰你必须碰的东西，只清理自己造成的混乱。

这是我认为最重要的一条工程习惯之一。

AI 非常喜欢“顺手优化一下”。它改一个 bug，可能顺带改格式、改命名、改相邻逻辑，甚至把它认为“不优雅”的旧代码一起重构掉。这样做的问题是，diff 会迅速膨胀，review 成本陡增，风险也同步上升。

所以我们要明确约束它：

- 不要改进相邻代码、注释和格式。
- 不要重构没有问题的代码。
- 即使做法不同，也尽量保持现有风格一致。
- 如果发现无关死代码，可以指出来，但不要擅自删除。

当 Agent 能做到精准修改时，团队对它的信任度才会真正提高。

### 4. 目标驱动执行

不要只给“做什么”，要给“什么算做成”。

这是我觉得最适合智能体时代的一条原则。因为 LLM 很擅长围绕一个明确目标持续循环：只要成功标准足够清晰，它就会不断尝试，直到满足条件。

比如不要只说“修复这个 bug”，而要说：

- 先写一个能复现 bug 的测试。
- 再让测试通过。
- 最后确认相关场景没有回归。

又比如不要只说“添加校验”，而要说：

- 为无效输入补测试。
- 让这些测试通过。
- 确认合法输入不受影响。

一旦你把任务转化成可验证目标，Agent 的自主性就会被真正激活。它不再需要你一句一句盯着，而是能沿着清晰的标准自己跑完整个闭环。

从这个角度看，“目标驱动执行”其实就是把“让它工作”这种模糊要求，升级成“让它通过这些可验证条件”的工程表达。

## 六、这对前端工程师意味着什么

最后我想讲一点偏感受层面的东西。

用了智能体之后，我会有一种很强烈的感觉：那种刚接触编程时的兴奋感，又回来了一部分。

因为以前很多“想做但嫌麻烦”的念头，现在真的有机会快速落地。你可以更快地验证一个交互想法，尝试一个页面重构方向，做一个自动化脚本，甚至把一些原本需要多个角色配合完成的事情，先自己跑出一个可用版本。

但另一方面，这也会逼着我们升级自己的能力结构。

我越来越认同这样一个判断：工程师的价值，正在从“代码产出速度”转向“决策质量”。

快速写代码这件事，未来会越来越不稀缺；但判断需求是否合理、设计方案是否成立、实现路径是否稳妥、生成结果是否可靠，这些能力会越来越值钱。

换句话说，未来更重要的不是“你能不能写”，而是“你能不能判断什么该写、怎么写、写到什么程度算够、以及结果到底好不好”。

这也是为什么我觉得，批判性思维、设计判断、架构理解、风险意识，这些原本就重要的能力，在 AI 时代只会继续升值，而且升值得更快。

如果借用一个不太严谨但很好懂的比喻：

LLM 很像一个掌握了很多数学定理的学生。它知道很多知识点，但刷题不够多，遇到复杂问题时，不一定知道这些定理该怎样组合使用。

而智能体做的事情，就像你在旁边不断给它提示：这一步先看条件，这一步先做验证，这一步再用哪个方法。于是它最终有机会把原本不会做的复杂题，真的做出来。

所以我并不觉得智能体会简单地“替代工程师”。更准确地说，它会重新定义工程师工作的重心。

代码本身会越来越便宜，但好判断、好约束、好流程、好验收，会越来越贵。

另外还有一个必须保持清醒的点是：AI 编码代理天然倾向于走最短路径。而最短路径，往往意味着跳过规范、跳过测试、跳过安全审查、跳过很多让软件可靠的工程实践。

所以我们不能只追求“它会不会做”，还要关注“它是不是按正确的流程在做”。

也正因为如此，Agent Skills 这样的机制会越来越重要。它不是在帮模型变聪明，而是在帮模型变得更像一个真正成熟的工程师，知道哪些步骤不能省，哪些标准必须过。

---

## 七、结尾

最后我想用三句话收一下今天的分享。

第一，AI 编程已经从代码补全，走到了对话副驾驶，再走到了智能体工作流时代。真正的变化，不只是模型更强了，而是它开始具备“行动能力”和“验证能力”。

第二，智能体之所以值得前端团队重点关注，是因为它和浏览器、视觉、交互、多端适配这些前端核心场景天然契合。尤其是 MCP 和 Skills 结合之后，它不只是能写代码，而是有机会参与完整的工程闭环。

第三，面对 AI，工程师最需要升级的，不是“如何提几个更花哨的提示词”，而是如何表达目标、约束流程、建立标准、做出判断。

如果说过去我们的竞争力，更多体现在“我能亲手写出多少代码”；那未来我们的竞争力，会越来越体现在“我能不能带着 AI，一起稳定地产出正确结果”。

我自己对这件事的态度其实很明确：它值得认真投入，而且越早建立方法论，收益越大。

谢谢大家，以上就是我今天的分享。
